# Demo Guide

A focused, presentation-ready walkthrough of **SentinelAI** — the
AI-driven intrusion detection and response dashboard. Optimized for a
~10-minute classroom slot.

For the long-form analyst walkthrough (every page, every field, every
quote-out-loud), see **[DEMO_SCRIPT.md](DEMO_SCRIPT.md)**. This guide is
the shorter sibling: setup, the story you're telling, the five things to
show, and what to do if something refuses to cooperate.

---

## 0. Startup (do this 5 minutes before class)

```bash
# Fresh laptop — one command brings up everything and seeds it:
make demo
```

`make demo` is the safe path for a live demo. Under the hood it runs:

1. **`make bootstrap`** — copies `.env`, brings up the three containers
   (`postgres`, `backend`, `frontend`), waits for `/health`, trains the
   detection model on 50k synthetic flows, and waits for the model to
   register.
2. **`make demo-seed`** — replays the bundled 60-flow sample, runs
   detection, marks one alert `CONFIRMED` (with an investigation packet
   and a generated markdown report), marks a second `UNDER_REVIEW`,
   approves one pending response action, rejects another, and finally
   produces today's `DAILY_SUMMARY` report.

When it finishes you'll see something like:

```
Dashboard is now seeded. Open: http://localhost:5173
  · Hero alert (CONFIRMED, with report):  /alerts/3
  · Second alert (UNDER_REVIEW):          /alerts/7
  · Reports page (with daily summary):    /reports
  · Response Center:                      /response
```

Open **three browser tabs** before class:

| Tab | URL                              | Why                                  |
| --- | -------------------------------- | ------------------------------------ |
| 1   | <http://localhost:5173>          | The dashboard you're presenting from |
| 2   | <http://localhost:5173/alerts/3> | The hero alert, pre-warmed           |
| 3   | <http://localhost:8000/docs>     | OpenAPI surface for the Q&A          |

A terminal window with `make logs-backend` open in the background is
useful if anyone asks "show me the audit trail" — every state change
prints a structured log line.

## 1. The story (~10 minutes)

You're a SOC analyst. Network traffic from a campus subnet streams in as
CSVs. The system **classifies** every flow, **prioritizes** the
suspicious ones, **recommends responses**, and writes **everything to an
audit chain** that proves what each agent (and each analyst) decided
and why. *All actions are simulated* — the system is a teaching
artifact, not a real firewall.

### Stage 1 — Dashboard (~1 min)

Open `/`.

- **Four KPI cards** — total events, suspicious events, open alerts,
  critical alerts. Demo seed leaves these non-zero (~60 events, ~30+
  alerts, several pending).
- **Alerts over time** stacked area chart — 24 h window, severity
  breakdown.
- **Severity donut + top attack categories** side-by-side.
- **Highest-priority alerts** table, with priority scores.
- **Model panel** — model name, version, threshold, feature count, class
  list. Proves the classifier is loaded and live.
- **Top-bar pills** (Backend / Database) both emerald = healthy.

> "One page that tells me the entire state of the SOC."

### Stage 2 — Alert detail (~3 min)

Click into the hero alert (or open Tab 2 directly).

Walk top to bottom:

1. **Header** with severity / status / disposition pills and src → dst
   IPs / port.
2. **Action bar** — two rows: disposition quick-actions (Confirm, Under
   review, False positive, Resolve) and workflow actions (Re-triage, Run
   investigation, Generate report, Close).
3. **Detection card** — predicted label, 4-decimal confidence, threshold,
   model identity, full class-probability table.
4. **Triage card** — severity, priority score, recent-flow count, and
   the rule lines (`family=BruteForce → criticality 0.70 × …`).
5. **Investigation card** — summary, bullet findings, eight statistics
   tiles, top contributing features. Pre-generated by demo seed.
6. **Related evidence** — related alerts (clickable IDs) and related
   events (chronological flows).
7. **Response actions table** — payload column, Approve / Reject buttons
   on PENDING rows. Click *Show payload* on one row to reveal the JSON
   that would be sent to a real firewall (it isn't).
8. **Decision chain** at the bottom — chronological `DETECTION → TRIAGE
   → RESPONSE → ANALYST → INVESTIGATION → ANALYST → REPORTING` rows.

> "Every state change writes one audit row. Nothing happens silently."

### Stage 3 — Response Center (~1 min)

`/response`.

- KPI cards: pending, auto-executed (24 h), approved, rejected.
- Pending queue with rationale column.
- "Simulated action execution history" table — every action ever fired,
  with the deciding analyst.

> "Every action row carries `simulated = TRUE`, enforced by a Postgres
> `CHECK` constraint. There is no code path in the project that contacts
> a real firewall."

### Stage 4 — Reports (~1 min)

`/reports`.

- Left: "Alerts without a report" + "All reports". A `DAILY_SUMMARY` is
  already present (demo seed).
- Right: the viewer renders the report markdown with full GFM tables and
  blockquotes. Two buttons: **Copy markdown** and **Download .md**.

Click the hero alert's report. Scroll the eight sections:

```
1. Incident Overview      2. Severity & Priority
3. Detection Results      4. Investigation Findings
5. Timeline               6. Response Recommendations
7. Analyst Action Status  8. Final Summary
```

> "Underneath, the report is structured JSON in a JSONB column. What
> you see rendered is what the database stores."

### Stage 5 — Ethics + audit, live (~30 s)

In a terminal:

```bash
make shell-db <<< "INSERT INTO response_actions (alert_id, action_type, simulated)
                   VALUES (1, 'BLOCK_IP', FALSE);"
```

→ `ERROR: new row for relation "response_actions" violates check
   constraint "ck_response_actions_simulated_only"`

> "Even raw SQL with admin credentials cannot record an action that
> claims it ran for real."

## 3. Fallback path (when live ingestion misbehaves)

The demo seed runs on a fresh database, so the Ingestion page on
`/ingestion` should already be empty of new work. If anyone asks to see
the live ingestion flow, use the three-step workflow on that page:

1. Click **Replay bundled sample** (single button, no file picker
   needed).
2. Wait for step 1 to turn green (~1 s).
3. Click **Run detection now** in step 2.
4. Click **Open Alerts console →** in step 3.

If any step fails, the fallbacks (in order) are:

### a. Re-run the smoke test from the terminal

```bash
make smoke
```

Eleven steps, each one printed with `✓` or `✗`. If it goes all-green
the system is fine and the failure was UI-side — refresh the browser.

### b. Bypass the UI entirely and curl the API

```bash
# Replay sample:
curl -fsS -X POST -H 'Content-Type: application/json' \
  -d '{"file":"samples/sample_flows.csv","rate":50}' \
  http://localhost:8000/api/v1/ingest/replay | jq

# Run detection:
curl -fsS -X POST -H 'Content-Type: application/json' \
  -d '{"limit":5000}' http://localhost:8000/api/v1/detection/run | jq
```

If those return JSON, walk through `/docs` (Tab 3) to show the API
surface — that's a perfectly fine alternative narrative.

### c. The pre-seeded state survives backend restarts

The demo seed lives in Postgres, not in container memory. If the backend
container crashes:

```bash
docker compose restart backend
```

…and the dashboard recovers without re-seeding. Only `make reset`
(which uses `docker compose down -v`) wipes the data.

### d. Worst case: re-bootstrap

```bash
make reset                   # wipe DB volume + restart
make demo-seed               # re-seed (uses the existing trained model)
```

This takes ~30 s end-to-end because the model is already trained.

### Report exports

On `/reports`, the right-hand viewer has two buttons:

- **Copy markdown** — paste straight into a write-up or Slack.
- **Download .md** — downloads `report-<id>.md`. To convert to PDF:

  ```bash
  pandoc report-3.md -o report-3.pdf \
      --pdf-engine=xelatex \
      -V geometry:margin=1in -V mainfont="Helvetica"
  ```

For the project submission, the **daily summary** report (auto-generated
by `make demo-seed`) is the best single document to attach — it
aggregates every alert in one place with no per-alert clicking.

### Recommended bundle for the term-paper appendix

1. The dashboard screenshot from above (#1).
2. The hero alert detail (#3 + #4 stitched, or two separate figures).
3. The markdown export of the hero report.
4. The markdown export of the daily summary.
5. The output of `make smoke` (proof that the pipeline works end-to-end).

## 5. Reset between back-to-back demos

```bash
make reset                  # wipes DB volume, brings stack back up
make demo-seed              # re-seeds in ~30 s (model artifact survives)
```

Or, if you only want to remove the demo state and keep the model + DB:

```bash
docker compose exec postgres psql -U sentinelai -d sentinelai -c \
    "TRUNCATE flow_events, alerts, response_actions, agent_decisions,
              reports, investigation_packets, ingestion_jobs RESTART IDENTITY CASCADE;"
make demo-seed
```

## 6. Optional: live sensor (authorized lab networks only)

> ⚠️ Only on a network you own or are explicitly authorized to monitor. Off by
> default; reads flow **logs** (Zeek/Suricata), never raw packets or payloads.

For an advanced demo you can show *real* flows arriving instead of CSV replay.
Easiest is `pcap_replay` against the bundled sample flow log — no Zeek/Suricata
install needed:

```bash
# 1. Get an analyst/admin token from your running stack
TOKEN=$(curl -fsS localhost:8000/api/v1/auth/login -H 'content-type: application/json' \
  -d '{"username":"admin","password":"<pw>"}' | jq -r .access_token)

# 2. (optional) auto-run detection after each batch for a live feel
#    set SENTINEL_DETECTION_AUTO_RUN_ON_INGEST=true on the backend, then restart it

# 3. Replay the sample flow log through the sensor (lab CIDRs cover the sample IPs)
cd sensor && pip install -e .
SENTINEL_SENSOR_ENABLED=true \
SENTINEL_SENSOR_MODE=pcap_replay \
SENTINEL_SENSOR_INPUT_PATH=./samples/conn.log \
SENTINEL_SENSOR_ALLOWED_CIDRS=192.168.0.0/16,10.0.0.0/8 \
SENTINEL_SENSOR_API_URL=http://localhost:8000 \
SENTINEL_SENSOR_API_TOKEN="$TOKEN" \
python -m sentinelai_sensor
```

Then show the Ingestion page's **Live sensor** card flip to *receiving*, and the
new flows turning into alerts. For a real Zeek/Suricata feed, point
`SENTINEL_SENSOR_MODE`/`INPUT_PATH` at your sensor's `conn.log`/`eve.json` and set
`ALLOWED_CIDRS` to your lab subnets. Full guide: [LIVE_SENSOR.md](LIVE_SENSOR.md).
